## Context

AIProxy 是一个多账户 LLM API 网关，当前代码库经过审计发现多个安全和业务逻辑问题：

**当前状态**:
- API 密钥在 `main.go:313` 直接存储原始值而非哈希
- 配置验证错误消息暴露 API 密钥值
- 限流器映射使用 provider name 而非 account ID，导致限流失效
- 重试机制在同一账户上重复尝试，浪费资源
- 限流窗口内存无限增长，无清理机制
- 服务重启后限流状态丢失

**约束条件**:
- 必须保持向后兼容，无 API 变化
- 修复必须不影响现有正常运行
- 需要处理多账户并发场景

**相关方**: 服务运维团队、安全团队

## Goals / Non-Goals

**Goals:**
- 修复所有安全漏洞（API 密钥泄露）
- 修复关键业务逻辑缺陷（限流失效）
- 实现限流状态持久化和恢复
- 实现智能账户切换重试
- 实现内存清理机制防止内存泄漏

**Non-Goals:**
- 本次不涉及架构重构（main.go 拆分另行处理）
- 不添加新功能特性
- 不改变现有 API 接口

## Decisions

### Decision 1: API 密钥存储方式

**选择**: 使用 SHA256 哈希存储 API 密钥

**理由**:
- 当前代码已有 `utils.HashAPIKey()` 函数
- SHA256 对高熵 API 密钥足够安全（虽无盐，但 API 密钥本身熵高）
- 保持与现有验证逻辑兼容

**备选方案考虑**:
- 加盐哈希: 增加复杂性，对高熵密钥收益有限
- 加密存储: 需要密钥管理，增加运维复杂度

### Decision 2: 限流器键映射修复

**选择**: 统一使用 account ID 作为限流器键

**实现方式**:
- 在 `initAccountLimiter()` 中按 account ID 存储
- 在 selector 中传入完整的 account -> limiter 映射
- selector.Select() 内部根据候选账户的 ID 获取对应限流器

**理由**:
- 每个账户应独立限流，不共享 provider 级别限流桶
- account ID 已有唯一生成逻辑（provider + key）

### Decision 3: 账户切换重试策略

**选择**: 失败后标记账户不可用，选择其他账户

**实现方式**:
- 在重试循环中调用 `pool.RecordFailure(account)`
- 每次重试重新调用 `selectAvailableAccount()` 获取新账户
- 使用已尝试账户列表避免重复选择同一账户

**理由**:
- 同账户重试无效（已触发限流或错误）
- 快速切换到可用账户提高成功率

### Decision 4: 内存清理机制

**选择**: 周期性清理过期限流窗口条目

**实现方式**:
- 在现有 cleanup ticker 中添加限流窗口清理
- 清理超过 1 小时未活跃的账户条目
- 在 RPM/Daily/Window 限流器中添加 `CleanupStale(maxAge time.Duration)` 方法

**理由**:
- 简单有效，不改变核心限流逻辑
- 与现有周期清理任务集成

### Decision 5: 限流状态恢复

**选择**: 服务启动时从数据库加载限流状态

**实现方式**:
- 在 `initAccountLimiter()` 中调用 `storage.GetRateLimit(accountID)`
- 将数据库中的 current_value 同步到内存计数器
- 仅恢复当日/当月的计数，滑动窗口无法精确恢复

**理由**:
- 数据库已有限流数据记录
- 避免重启后限流"重置"

## Risks / Trade-offs

### Risk 1: 限流状态恢复不精确
- **风险**: 滑动窗口（RPM）无法精确恢复时间戳
- **缓解**: 恢复总计数而非时间戳，接受短暂误差；5 分钟后自然收敛

### Risk 2: 账户切换可能导致所有账户不可用
- **风险**: 所有账户同时触发限流时无法重试
- **缓解**: 返回 429 错误，等待限流窗口恢复；此为预期行为

### Risk 3: 内存清理可能清理活跃账户
- **风险**: 1 小时阈值可能在低频使用场景清理活跃账户
- **缓解**: 使用账户最后使用时间而非最后清理时间；可配置阈值

### Risk 4: 配置文件 API 密钥移除影响部署
- **风险**: 用户依赖 config.json 中的密钥配置
- **缓解**: 提供环境变量示例文档；支持两种方式过渡

## Migration Plan

### 部署步骤
1. 部署修复版本，API 密钥哈希存储生效
2. 重启服务后限流状态自动从数据库恢复
3. 内存清理随周期任务自动执行

### 回滚策略
- 如限流恢复有问题，可禁用恢复逻辑（配置项）
- 如账户切换有问题，可回退到原重试逻辑

## Open Questions

- 内存清理阈值（1 小时）是否合适？建议可配置
- 限流恢复是否需要处理跨天/跨月边界？当前依赖自然重置
